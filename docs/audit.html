What Was Checked

Each commit within the audit window was verified against the following criteria:

  • Commit signing: Every commit must have a valid GPG or SSH signature
  • PR traceability: Every commit must be the merge result of an approved pull request (no direct pushes)
  • CI integrity: Required status checks must have passed before merge (per branch protection / rulesets)
  • No post-merge tampering: No commits pushed to branches after their PR was merged or closed
  • Commit message originality: No commit messages replicated from earlier legitimate commits (social engineering indicator)
  • Dependency provenance: New/updated dependencies checked against registry for release date and yank status
  • Renovate cooldown enforcement: Each dep's adoption timing verified against the repo's configured minimumReleaseAge policy (violations are critical)
  • Known vulnerability scan: All current packages (from lock files) checked against OSV.dev for disclosed CVEs and advisories
  • Branch protection stability: No modifications to branch protection rules or rulesets during the window
  • Post-approval PR integrity: All commits within each PR branch inspected for pushes after the last review approval — especially from unexpected authors
  • Human review requirement: Every merged PR must have at least one approval from a human reviewer (bot-only approvals flagged)
  • Squash/merge via GitHub UI: Accepted as legitimate when they are the merge commit of a reviewed PR

Security Recommendations

Prioritized actions based on this audit's findings, ordered by impact.

1. Upgrade pillow across 7 repos to fix 31 known vulnerabilities
31 of 37 findings are known vulnerabilities, primarily pillow@12.2.0 (CVEs PYSEC-2026-2253 through 2257) affecting ansible-compat, ansible-creator, ansible-dev-environment, ansible-dev-tools, ansible-lint, pytest-ansible, tox-ansible, and vscode-ansible. This is the single highest-impact action: upgrade pillow to the patched release across all affected repos. Coordinate via a single Renovate-driven batch or manual PR to each repo. Track closure in a shared issue.
2. Eliminate orphan commits: require all changes go through PRs
2 high-severity orphan commits were pushed directly to vscode-ansible main without a PR (vault refactor commits by sathyapramod). Direct pushes bypass code review, CI gating, and audit trails. Enable the Restrict pushes that create matching refs ruleset rule on all 12 repos so that only merge-queue or PR merges can land on the default branch. Admins should not be exempt.
3. Require commit signing across all repos (not just ansible-dev-environment)
Only ansible-dev-environment enforces required signatures. The other 11 repos allow unsigned commits, meaning a compromised credential could push unverifiable code. Enable Require signed commits in the GitHub ruleset for all repos. Pair this with onboarding docs so contributors set up GPG or SSH signing before their first PR.
4. Block post-approval commits or require re-review
1 high-severity finding: ansible-navigator PR #2120 had commits pushed after approval by ansibuddy. Post-approval commits can silently alter reviewed code. Enable Dismiss stale pull request approvals when new commits are pushed in the required reviews ruleset for all 12 repos. This forces a fresh review whenever new code lands after approval.
5. Require at least one human reviewer for bot-authored PRs
ansible-navigator PR #2120 was approved only by ansibuddy (a bot). While this was a dependency update (low risk), bot-only approvals set a dangerous precedent. Configure CODEOWNERS or branch protection to require at least 1 human approval on all PRs, including those authored by Renovate or Dependabot. Treat bot approvals as informational, not sufficient.
6. Enforce admin compliance with branch protection
Only molecule has enforce_admins=True. The other 11 repos allow administrators to bypass all protections (force push, skip reviews, skip CI). Enable Do not allow bypassing the above settings in rulesets for all repos. Admins with bypass privileges represent the highest-privilege attack surface and should be subject to the same controls.
7. Standardize Renovate cooldown policies
vscode-ansible uses a 1-day cooldown with no major-version delay, while the other 10 repos (via shared config) use 2 days / 7 days for major bumps. ansible-builder has no Renovate config at all. Standardize all repos on the shared ansible/actions//config/renovate.json config and ensure minimumReleaseAge is at least 3 days (ideally 7 for major) to allow time for community detection of compromised releases.
8. Investigate and stop GitHub-web-signed non-merge commits
2 commits in vscode-ansible were signed by GitHub's web key (not a personal key) and are not PR merge commits. This typically means code was edited directly in the GitHub web UI and pushed to main. This pattern bypasses local tooling, pre-commit hooks, and may indicate a workflow shortcut. Establish a team norm: never use the GitHub web editor for code changes to the default branch. Use branch protection to block direct pushes (see recommendation #2).
9. Run automated vulnerability scanning in CI, not just at audit time
The 31 known vulnerabilities were only detected during this audit. Integrate osv-scanner or pip-audit as a required CI check on all repos so vulnerable dependencies block merges in real time. This shifts detection left and prevents accumulation of known-vulnerable packages between audits.
10. Establish a recurring supply chain audit cadence
This audit covered a single week and found 37 findings including 3 high-severity issues. Schedule this audit to run weekly (ideally automated via CI cron) with results posted to the team's security channel. Assign a rotating security champion each week to review the report, triage new findings, and ensure prior recommendations are being implemented. Track open findings as issues, not just report entries.
